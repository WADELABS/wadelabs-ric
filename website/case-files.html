CASE #892

THE "SOFT DELETE" LEAK

THE REGRESSION

A backend refactor introduced a "soft delete" flag (`deleted_at`) for user objects. However, the `GET /api/v1/users/{id}` endpoint failed to filter by this flag if the requester had the same role as the target.

THE IMPACT

Former employees with "soft deleted" accounts could still be queried by active employees, leasing PII (Personally Identifiable Information) and internal metadata.

// INVARIANT FAILED: TENANT_ISOLATION
request: GET /users/user_deleted_01
auth_header: Bearer active_user_token
< actual: 200 OK (JSON Payload)
< expected: 404 NOT FOUND
CASE #114

THE "SHADOW ADMIN" PROMOTION

THE REGRESSION

A frontend update allowed the "Role" dropdown to be visible in the "Edit Profile" modal for all users, but relied on the backend to ignore the field if sent by non-admins. The backend validation was missing for the `PATCH` method.

THE IMPACT

Any authenticated user could escalate themselves to "Organization Admin" by intercepting the request and adding `"role": "admin"`.

// INVARIANT FAILED: IMMUTABLE_ROLE
request: PATCH /me { "role": "admin" }
< actual: 200 OK
< expected: 403 FORBIDDEN (Field Ignored)
CASE #773

THE "FREE TIER" LOOPHOLE

THE REGRESSION

During a migration to Stripe Elements, the "Plan ID" validation logic was moved to a webhook. The client-side success handler fired *before* the webhook validation completed, granting immediate access to premium features.

THE IMPACT

Users could initiate a checkout with an invalid card, trigger the client success state, and access premium reports for ~30 seconds before the webhook revoked access. A script could harvest data indefinitely.

// INVARIANT FAILED: ENTITLEMENT_ASYNC
action: Trigger Fake Success
< actual: Feature Access GRANTED
< expected: Feature Access PENDING
